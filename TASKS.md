# Expense Tracker — Build Tasks

Offline-first means: every write hits SQLite first, UI is always reactive to local state, and the sync engine pushes/pulls in the background when online. The backend never blocks the UI.

---

## Architecture decisions

- **Local DB is source of truth.** All reads come from SQLite via Drizzle. No loading spinners waiting for the network.
- **Sync via change log.** Every mutating operation appends to a `sync_queue` table. The sync engine drains the queue when online.
- **Soft deletes everywhere.** Rows are never hard-deleted locally — `deleted_at` timestamp acts as a tombstone so deletes can be synced.
- **Conflict strategy: last-write-wins on `updated_at`.** Simple enough for a single-user app. Re-evaluate if multi-device conflicts become a real problem.
- **TanStack Query owns the sync layer.** React Query mutations write to SQLite, then trigger a background sync. Query cache is hydrated from SQLite, not from the network directly.

## Database schema (`src/db/schema.ts`)

```
users           id, email, name, created_at
categories      id, name, icon, color, is_default, deleted_at, updated_at
expenses        id, amount_cents, currency, category_id, note, date, receipt_url,
                created_at, updated_at, deleted_at, synced_at
budgets         id, category_id, limit_cents, period (monthly|weekly), starts_at,
                updated_at, deleted_at
sync_queue      id, operation (create|update|delete), entity_type, entity_id,
                payload (JSON), created_at, attempts, last_error
```

All IDs are client-generated UUIDs so records can be created offline without a server round-trip.

### Offline-first sync engine (`src/lib/sync-engine.ts`)

- [ ] **Queue writes** — every SQLite mutation appends to `sync_queue`
- [ ] **Drain queue** — on network reconnect and on app foreground, POST `/sync/push` with pending ops
- [ ] **Pull remote changes** — after push succeeds, GET `/sync/pull?since=<last_synced_at>` and upsert into SQLite
- [ ] **Conflict resolution** — `conflict-resolver.ts` compares `updated_at`, keeps newer record
- [ ] **Retry with backoff** — failed ops stay in queue, retry up to 5 times with exponential backoff; surface persistent errors in SyncIndicator
- [ ] **Tombstone propagation** — deleted records sync `deleted_at` to server so other devices remove them

## Backend integration points (FastAPI)

Routes the sync engine talks to:

```
POST   /auth/login
POST   /auth/register
POST   /auth/refresh

GET    /sync/pull?since=<iso_timestamp>    # Returns changed records since timestamp
POST   /sync/push                          # Accepts array of sync operations
```

All entity CRUD endpoints should also exist for initial full sync on new device login:

```
GET    /expenses
GET    /categories
GET    /budgets
```

## Performance notes

- Use `@shopify/flash-list` for all scrollable lists — not `FlatList`
- Aggregation queries (analytics) run in SQLite — do not compute in JS
- Drizzle queries are synchronous in expo-sqlite v2 — keep repositories thin, no heavy logic
- Memoize chart data derivations with `useMemo`
- Avoid storing large blobs (receipt images) in SQLite — write to `expo-file-system` and store the URI
